## Detailed Failure Analysis

### Llama 3.3 70B

- **Outcome:** Exit 0, viz_spec written, but code is 45 chars
- **Code produced:** `window.EXPLAINER_VIZ = (function() {... })();`
- **Root cause:** Model used JavaScript spread-rest `...` as a literal placeholder instead of implementing the function body. Orchestrator's Node.js syntax check caught `SyntaxError: Unexpected token '...'` and retried — second attempt identical.
- **Likely cause:** Model treated the task as "fill in placeholder" rather than full implementation. May also be a quality/instruction-following limitation at 70B.

### Qwen 2.5 Coder 32B

- **Outcome:** HTTP 400 from OpenRouter
- **Error:** `maximum context length is 32768 tokens. However, you requested about 60429 tokens (44429 of text input, 16000 in the output)`
- **Root cause:** viz_worker.md prompt (including lesson plan context) is ~44K tokens, exceeding Qwen's 32K provider context window.
- **Infrastructure gap:** Stage 4 prompt cannot be shortened below ~35K tokens without degrading output quality. Models need ≥64K context for Stage 4.

### Mistral Small 3.1 24B

- **Outcome:** `json.decoder.JSONDecodeError: Unterminated string starting at: line 4 column 15 (char 55)`
- **Raw response:** `{"mode": "custom_code", "config": {"plugin": "nested_circles_v`
- **Root cause:** Unknown — either model hit a generation cutoff, OpenRouter truncated the response, or the model returned an incomplete JSON object. Retried once with same result.
- **Infrastructure gap:** Mistral Small's JSON output mode appears unreliable for large structured responses.

### Claude Sonnet 4 (direct)

- **Outcome:** HTTP 401 authentication error
- **Root cause:** ANTHROPIC_API_KEY stored in `~/.zshrc` was revoked between prior pipeline runs and this benchmark session. OpenRouter key also not found in shell environment.
- **Note:** The bench_claude baseline uses a gpt4omini_v2 viz artifact from a prior run (same problem, same declared actions).

## Infrastructure Requirements for Cheap Model Support

To run Stage 4 with cheaper models, the following must be addressed:

1. **Context window ≥ 64K tokens required** — viz_worker.md + lesson plan + act specs is ~44K tokens input. Models with 32K provider limits (Qwen coder series) cannot be used.

2. **JSON output mode must be reliable** — Stage 4 returns a JSON object. Models that truncate or hallucinate JSON (`{"mode": ...}` cut at 55 chars) need either a retry strategy with smaller max_tokens or a different prompting approach.

3. **No placeholder generation** — Llama 3.3 70B treated `...` as valid output. Prompt engineering fix: viz_worker.md now explicitly forbids `//` comments and requires complete implementations. A stronger guard against placeholder code may be needed.

4. **Recommended models for Stage 4** (≥64K context, reliable JSON):
   - `openrouter:anthropic/claude-3-5-haiku` (context: 200K)
   - `openrouter:meta-llama/llama-3.1-405b-instruct` (context: 128K)
   - `openrouter:google/gemini-flash-1.5` (context: 1M)
   - `openrouter:deepseek/deepseek-coder` (context: 128K)

## Known Bug Documented

The `_normalize_viz_code()` function in `assembler.py` inserts `\n` after `;`, `{`, `}` in single-line JSON-embedded JavaScript. This correctly terminates `//` comments at statement boundaries. However, if a `//` comment appears in the MIDDLE of a multi-part statement (e.g., `for (var r = 1; // comment   r <= N; r++)`), the comment will capture part of the statement.

**Mitigation in prompt:** viz_worker.md now explicitly bans `//` comments, requiring `/* */` block comments instead. New model outputs should not trigger this bug.
